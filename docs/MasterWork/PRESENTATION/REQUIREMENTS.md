## Слайд 4. Выбор подхода и ключевые решения

**Заголовок:**
Обоснование выбранных решений

**Основные тезисы на слайде:**
- Для детекции выбран одноэтапный детектор семейства YOLO
- Для сопровождения выбран трекер класса DeepSORT
- Для событийного анализа выбран правиловой подход
- Для интеграционного контура выбран RabbitMQ + transactional outbox
- Для хранения структурированных данных выбран PostgreSQL, тяжелые артефакты вынесены в файловый слой

**Почему это важно озвучить:**
- Выбор сделан по критерию баланса между точностью, задержкой, интерпретируемостью и простотой внедрения
- Работа посвящена не только распознаванию, но и сквозному эксплуатационному контуру
- Отдельным блоком нужно явно сформулировать научную новизну: согласование real-time CV-конвейера `capture -> detect -> track -> rules` с надежной событийной доставкой `outbox -> RabbitMQ/webhook`
- Формулировка новизны должна быть конкретной: не «использованы современные методы», а «предложен и апробирован сквозной контур с `latest-frame`, `dedup_key`, transactional outbox, retry и DLQ`

## Слайд 6. Алгоритмический конвейер и правила событий

**Заголовок:**
Алгоритмический конвейер платформы

**Основные тезисы на слайде:**
- Конвейер: `capture -> detect -> track -> rules -> outbox -> RabbitMQ/webhook`
- Реализованы правила:
  - `zone_enter`
  - `line_cross`
  - `dwell_time`
- Для подавления повторов используется `dedup_key`
- Для режима реального времени применяется latest-frame очередь и ограничение накопления задержки

**Что подчеркнуть устно:**
- Платформа формирует не просто детекции, а интерпретируемые события
- События надежно проходят путь до внешнего потребителя

## Слайд 10. Полученные результаты

**Заголовок:**
Основные результаты апробации

- Общий итог прогона: `result = passed`, `duration_sec = 30.732`
- Security:
  - `viewer_create_camera_status = 403`
- Functional E2E:
  - `outbox_status = published`
  - `retry_count = 1`
  - `delivery success = 1`, `failed = 0`
  - `functional_delivery = 4.605 c`
- RabbitMQ reliability:
  - во время отказа `status = retry`, `retry_count = 1`
  - после восстановления `status = published`
  - `delivery success = 1`, `failed = 0`
  - `rabbitmq_recovery = 8.863 c`
- Webhook retry:
  - `failed_attempts_before_recovery = 1`
  - `delivery success = 1`, `failed = 2`
  - `outbox_status = published`
  - `webhook_recovery = 4.455 c`
- Ограничение текущего этапа:
  - промышленная многокамерная p95-оценка не заявляется как достигнутый результат
