# Речь к презентации магистерской работы

Расчетная длительность: 7-10 минут.

## Слайд 1. Титульный

Уважаемый председатель, уважаемые члены государственной экзаменационной комиссии.

Представляю магистерскую работу на тему: «Разработка и экспериментальная оценка платформы видеоаналитики реального времени для обнаружения, сопровождения и событийного анализа объектов в системах охранного видеонаблюдения».

В работе рассматривается не только модель компьютерного зрения, а полный прикладной контур: видеопоток, детекция, трекинг, правила событий, надежная доставка, операторский интерфейс и проверяемая приемка.

## Слайд 2. Проблема

В охранном видеонаблюдении растет число камер и объем видеоданных. Оператору сложно устойчиво контролировать большое количество потоков без пропуска событий.

Практической системе недостаточно просто обнаружить объект на кадре. Нужно понять, какое событие произошло, сохранить доказательные данные, показать их оператору и доставить уведомление во внешний контур без потери при кратковременных сбоях.

Отдельный пример - беспилотные летательные аппараты: они могут быть малозаметными, динамичными и появляться на короткое время.

## Слайд 3. Цель и задачи

Цель работы - повысить эффективность автоматизированного мониторинга охраняемых зон за счет разработки и экспериментальной оценки платформы видеоаналитики реального времени.

Для достижения цели были сформированы требования, выбран стек методов детекции и трекинга, разработаны архитектура и алгоритмический конвейер, реализована программная платформа, подготовлен БПЛА-демо-сценарий и проведена экспериментальная проверка полного E2E-контура.

## Слайд 4. Ключевые решения

В качестве базовых решений выбраны YOLO для детекции, DeepSORT-подход для сопровождения, правиловой слой для событий `zone_enter`, `line_cross`, `dwell_time` и transactional outbox для надежной доставки.

Научная и инженерная новизна состоит в согласовании real-time CV-конвейера `capture -> detect -> track -> rules` с событийной архитектурой `events -> outbox -> RabbitMQ/webhook`, где результат оценивается не только метриками модели, но и сохранностью событий.

## Слайд 5. Архитектура

Платформа разделена на несколько сервисов. `api` отвечает за конфигурацию, авторизацию, журнал событий, подтверждение инцидентов и экспорт evidence. `analytics-worker` обрабатывает видео, выполняет детекцию, трекинг и применяет правила. `integration-worker` публикует события и выполняет повторные попытки доставки.

PostgreSQL хранит конфигурацию, события и outbox, RabbitMQ используется как транспорт, Prometheus и Grafana обеспечивают наблюдаемость, а React UI служит рабочим местом оператора.

## Слайд 6. Алгоритмический конвейер

Сквозной путь начинается с приема RTSP или MP4-потока. Для ограничения накопления задержки используется latest-frame политика: при перегрузке важнее обработать свежий кадр, чем догонять старую очередь.

Далее выполняется детекция объектов, сопровождение треков, применение правил и запись события в базу. После этого событие попадает в outbox и доставляется отдельным интеграционным воркером. Такое разделение снижает риск потери события при сбое внешнего компонента.

## Слайд 7. Реализация

Backend реализован на FastAPI, SQLAlchemy, Alembic и Pydantic. Контур доступа использует JWT и RBAC для ролей `admin`, `operator` и `viewer`.

Frontend реализован на React, TypeScript и Vite. Для оператора доступны рабочие маршруты `/dashboard`, `/demo`, `/events`, `/cameras`, `/zones`, `/rules` и `/status`. Экспериментальный маршрут с вариантами frontend исключен из финальной навигации.

## Слайд 8. UI и демо

На слайде показаны ключевые экраны: сводная панель, БПЛА-демо, журнал событий и статус сервисов.

В журнале событий оператор может подтвердить инцидент через backend mutation `PATCH /api/v1/events/{event_id}/ack` и выгрузить доказательный JSON через `GET /api/v1/events/{event_id}/evidence`. Это важно, потому что действия UI не остаются локальным состоянием браузера, а фиксируются в backend.

## Слайд 9. Методика апробации

Апробация выполнена в CPU-first контуре Docker Compose. Проверялись функциональные E2E-сценарии, отказ RabbitMQ, недоступность webhook-получателя, восстановление доставки, RBAC/JWT и демонстрационный сценарий обнаружения БПЛА.

Основные метрики и признаки приемки: общий статус acceptance, успешная доставка событий, отсутствие потерь в кратковременных отказах, `DLQ = 0` в штатном прогоне, запрет write-операции для `viewer` со статусом 403.

## Слайд 10. Результаты CPU acceptance

Acceptance-прогон завершился со статусом `passed`, длительность составила 30.732 секунды.

Функциональная доставка события прошла успешно: outbox получил статус `published`, успешная доставка равна 1, ошибок доставки в базовом сценарии нет, время functional delivery составило 4.605 секунды.

При отказе RabbitMQ событие осталось в outbox в статусе `retry`, после восстановления было опубликовано, время восстановления составило 8.863 секунды. При недоступности webhook доставка также восстановилась, время восстановления составило 4.455 секунды. RBAC подтвержден: роль `viewer` получила 403 при попытке создать камеру.

## Слайд 11. Выводы и развитие

В результате разработана и апробирована платформа видеоаналитики реального времени, объединяющая детекцию, трекинг, событийные правила, надежную доставку уведомлений и операторский интерфейс.

Практическая значимость состоит в том, что решение можно использовать как воспроизводимый исследовательско-экспериментальный стенд для задач охранного видеонаблюдения.

Дальнейшее развитие связано с расширением датасета БПЛА, межкамерной re-identification, объектным хранилищем для артефактов, кластеризацией инфраструктуры и аппаратным ускорением инференса через ONNX Runtime или TensorRT.

Поставленная цель магистерской работы достигнута. Благодарю за внимание.

## Возможные вопросы комиссии

### В чем научная новизна?

Новизна состоит в согласовании алгоритмического конвейера видеоаналитики с архитектурой надежной событийной доставки. Система оценивается не только по качеству детекции, но и по сквозным эксплуатационным свойствам: задержке, сохранности событий, retry, DLQ и RBAC.

### Почему выбран правиловой событийный анализ?

Для охранных сценариев важна интерпретируемость. Правила `zone_enter`, `line_cross` и `dwell_time` понятны оператору, проверяемы в тестах и хорошо сочетаются с трекингом.

### Почему нужен transactional outbox?

Outbox разделяет фиксацию события в БД и публикацию во внешний контур. Если брокер или webhook недоступны, событие остается в базе и может быть доставлено после восстановления.

### Что ограничивает промышленное применение текущей версии?

Нужны более разнообразные объектовые данные, длинные многокамерные прогоны, межкамерная корреляция, отказоустойчивая инфраструктура PostgreSQL/RabbitMQ и дополнительная оптимизация инференса под целевое оборудование.
